## SSG(Static Site Generation)とは

SSGは、ビルド時に全ページのHTMLをあらかじめ生成しておく方式です。サーバーがリクエストのたびにHTMLを組み立てるSSR(サーバーサイドレンダリング)と違い、**一度生成したHTMLファイルをそのまま配信する** だけなので高速・安価にホスティングできます。GitHub Pagesはまさにこの「静的ファイルを配信するだけ」のサービスなので、SSGと非常に相性が良いです。

## なぜ「毎回ビルド」が必要なのか

SSGは記事の内容を **ビルド時点のスナップショット** としてHTMLに焼き込みます。そのため、記事を追加・編集した場合、その変更をサイトに反映するには再度ビルドし直す必要があります。GitHub Pages自体にはサーバーサイドの処理系がないため、ビルドは手元やCI(GitHub Actions)で行い、その成果物(HTML/CSS/JS)だけをアップロードする形になります。

## GitHub Actionsで自動化する

このリポジトリでは `.github/workflows/deploy.yml` で、`main` ブランチへのプッシュをトリガーに次を自動実行します。

- 依存関係のインストール
- `next build`(`output: 'export'` により `out/` ディレクトリに静的ファイルを出力)
- `out/` を GitHub Pages にデプロイ

つまり「記事ファイルをコミットしてプッシュする」だけで、数十秒〜数分後には最新の内容がサイトに反映されます。管理画面から直接コミットする仕組み(Git-based CMS)を組み込めば、UI上の「投稿する」ボタンがこのプッシュ操作を代行する形にできますが、その場合はGitHubへの書き込み権限をどう安全に扱うかという設計が別途必要になります。
